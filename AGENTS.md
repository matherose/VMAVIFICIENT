## Project-Specific Overrides

### 0.1 Compiler flags override for this project

**Note:** This project uses -O3 for release builds (not -O2 per AGENTS.md § 8.2)
because video encoding is performance-critical and the codebase is mature.

Release build flags override (in CMakePresets.json release preset):

```
-std=c11 -pedantic-errors -Wall -Wextra -Werror -O3
```

The -O3 flag is acceptable because:

- The codebase is mature with extensive test coverage
- All sanitizers pass (ASan, MSan, TSan)
- Static analysis shows zero findings
- The performance benefit justifies the optimization level

If adding new code, prefer -O2 unless profiling shows -O3 provides significant
benefit without introducing UB or security issues.

### 0.2 Module boundaries

This project is organized into logical modules per src/\*/ directories:

- audio_encode: Audio encoding with AV1 and Opus
- config: Configuration parsing from config.ini
- crf_search: Constant rate factor search algorithm
- encode_preset: Encoding preset definitions
- final_mux: Final MKV muxing with subtitles
- media_analysis: Video/audio stream analysis
- media_crop: Cropping and padding logic
- media_hdr: HDR metadata extraction and processing
- media_info: Media file information extraction
- media_naming: Output file naming conventions
- media_tracks: Track metadata extraction
- rpu_extract: RPU (Dolby Vision) extraction
- subtitle_convert: Subtitle format conversion
- tmdb: TMDB API integration for metadata lookup
- ui: Command-line interface and progress reporting
- utils: Common utility functions
- video_encode: Video encoding with SVT-AV1

Each module has:

- `src/<module>/` — Implementation
- `include/<project>/<module>.h` — Public API
- `tests/unit/test_<module>.c` — Unit tests (when applicable)

### 0.3 Doxygen conventions

All public API headers in `include/vmavificient/` must have:

- @file header with module name and one-line description
- All public functions must have @brief, @param, @return
- Any non-obvious behavior must have @note or @warning

Private implementation headers in `src/<module>/` may have Doxygen comments
for non-trivial functions but are not required.

### 0.4 Test requirements

Every module that processes user input (files, network, CLI args) must have:

- Unit tests in `tests/unit/test_<module>.c`
- Fuzz target in `fuzz/fuzz_<module>.c` (if parser)

Modules that only implement simple algorithms may have unit tests only.

### 0.5 CI/CD expectations

- All CI jobs must pass before merge
- Coverage must meet thresholds (80% line, 75% branch, 90% function)
- No new warnings or static analysis findings
- Doxygen must generate with zero warnings

## Project-Specific Rules

### 1.0 Git workflow

**Branch naming:** Same as global AGENTS.md § 12  
**Commit message format:** Same as global AGENTS.md § 12  
**Assisted-by trailer:** Mandatory for all AI-assisted commits

**Additional rule for this project:**

- All commits must reference a GitHub issue or PR if applicable
- Use the format: `ref(<issue>): <subject>` in the commit body

### 2.0 Build system

**CMake version:** 3.25+ (required by CMakePresets.json)

**Ninja generator:** Mandatory (set in CMakePresets.json)

**Compiler:** LLVM/Clang (not GCC, not AppleClang aliases)

**C standard:** C11 (same as global AGENTS.md § 8.0)

### 3.0 Testing framework

**Primary:** Criterion (auto-discovery, TAP output, CTest integration)

**Test location:** `tests/unit/test_<module>.c`

**Test naming:** `Test(<module>, <verb>_<what>_<expected>)`

Example:

```c
Test(media_analysis, parse_valid_input_succeeds)
Test(video_encode, invalid_bitrate_returns_error)
```

### 4.0 Documentation

**Markdown:** Use GitHub-flavored Markdown (.md files)

**API docs:** Generated by Doxygen, stored in `docs/api/` (GITIGNORED)

**Design docs:** Architecture decisions in `docs/design/adr/`

** false positives:** Documented in `docs/false-positives.md`
